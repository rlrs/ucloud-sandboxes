# Sandbox service implementation audit

This is a source-and-test audit of the repository as of 2026-07-10. “Fixed”
means the invariant is implemented and covered by repository tests; it does
not imply a multi-host fault-injection or production load test. The system is
currently strongest as a single-host control-plane deployment whose processes
share local POSIX locks and SQLite files.

Status terms:

- **Fixed in this work** — implementation and regression coverage are present.
- **Mitigated** — the common failure is fenced, but an operational or topology
  boundary remains.
- **In progress** — the state-machine work exists, but end-to-end rollout or
  deployment wiring is not complete.
- **Remaining** — no complete implementation-level fix is present.

## Correctness and distributed correctness

| ID | Severity | Failure mode | Current status | Relevant components |
|---|---|---|---|---|
| C1 | Critical | A delayed create/delete for a reused sandbox ID could mutate a newer incarnation (ABA), or a gateway crash could lose which operation had run. | **Fixed in this work.** Routes persist a never-reused generation high-water, create/delete operation IDs, canonical spec hash, node epoch and activity epoch. The node enforces idempotent replay, stale-generation rejection and durable tombstones. | `routing.py`, `control_plane.py`, `sandbox.py`, `node_agent.py` |
| C2 | Critical | A runtime container could be created before node state was committed, leaving an orphan after a crash; concurrent processes could lose records or reuse SSH ports. | **Fixed in this work.** Cross-process state locking, monotonic revisions, runtime identity labels, runtime-side recovery, atomic tombstones and concurrent port allocation are covered. | `sandbox.py`, `node_agent.py` |
| C3 | Critical | An ambiguous node create could be retried on another node, or stale/empty inventory could overwrite or delete a newer route. | **Fixed in this work** for protocol-aware nodes: create intent is persisted before dispatch and replayed only with the same node/generation/operation/spec; responses and complete inventory are identity-checked; an omission is accepted only from complete inventory for the same node epoch after the route activity barrier. Creating/unknown routes and unordered observations remain reserved rather than being relocated. | `control_plane.py`, `routing.py`, `models.py` |
| C4 | Critical | Delete timeouts, non-2xx replies or delayed replies could remove the local route while the sandbox remained live, or delete a later generation. | **Fixed in this work.** A stable delete operation and generation headers are persisted and reused; the route and persistent registry reference remain on ambiguous/non-2xx results and are removed only by a matching 2xx result. | `control_plane.py`, `routing.py`, `node_agent.py`, `sandbox.py` |
| C5 | Critical | Two gateway processes could both place against the same free capacity, or a persisted route plus its process-local reservation could be charged twice. | **Fixed in this work for same-host processes sharing local POSIX locks/state.** A cross-process placement lock spans selection through durable intent persistence, and matching route/in-flight identities are de-duplicated. **Remaining for multi-host deployments** without a consensus/transactional placement store. | `control_plane.py`, `routing.py` |
| C6 | Critical | Multiple autoscalers could both mutate UCloud; a crash between provider submission and local state update could duplicate VMs or stops. | **Fixed for the supported single-control-host topology.** A process-lifetime POSIX lock admits one mutating controller, while a compact SQLite journal records prepared/uncertain/accepted/settled/failed operations. Creates carry stable provider-operation labels and unresolved operations are reconciled rather than blindly replayed. Multi-host leadership requires an external transactional coordinator. | `autoscaler_state.py`, `cli.py`, `reconcile.py`, `ucloud.py` |
| C7 | High | The autoscaler could plan from only the first UCloud page, undercount stale/incompatible jobs, exceed hard node/provisioning limits, or loop on a cursor. | **Fixed in this work.** Job browsing is exhaustive and fails closed on malformed/repeated cursors; all non-final jobs count against hard limits even when they receive no capacity credit. | `ucloud.py`, `policy.py`, `reconcile.py` |
| C8 | High | A provider create acknowledged by UCloud but not yet visible in browse results could trigger a replacement; partial scale-up could consume a larger demand signal. | **Fixed in this work** for visibility and demand consumption: accepted creates hold a visibility guard until an exhaustive inventory observes their target job IDs, then settle; demand is consumed only when accepted/recovered operations cover the planned deficit. **Mitigated:** an uncertain create with no unique label correlation, duplicate correlations, or indefinite provider invisibility requires operator resolution and deliberately blocks replacement. | `cli.py`, `autoscaler_state.py`, `reconcile.py` |
| C9 | High | Provider stop ambiguity could remove heartbeats/routes or be replayed unsafely. | **Fixed in this work.** Stop intent is journaled against one immutable job ID. Final provider inventory recovers it without another call; otherwise the same operation is made retryable, but execution is allowed only after the current cycle re-posts the durable drain token and a fresh heartbeat proves that exact token ready. Heartbeats/routes are removed only after definite success or final-state recovery. | `cli.py`, `autoscaler_state.py` |
| C10 | Critical | Scale-down based on an idle boolean or stale zero count could stop a VM while work or reservations still existed; an obsolete durable drain could terminate capacity after demand rose. | **Fixed in this work.** A durable drain token closes node create/build admission, and the provider stop is journaled/executed only when a fresh complete heartbeat acknowledges the same token and activity epoch with zero inventory, work and reservations. Stable drain intent survives controller restart in the recurring loop and `--once` mode. Active drains are counterfactually replanned; newly needed capacity transitions durably to `canceling` before an exact-token undrain, and ambiguous cancellation can never authorize termination. | `sandbox.py`, `node_agent.py`, `agent.py`, `autoscaler_state.py`, `cli.py`, `reconcile.py` |
| C11 | High | A sender-controlled/future heartbeat timestamp could keep a dead node fresh; cross-deployment heartbeats could be accepted; idle time could begin on the sender; truthy strings or a filtered malformed inventory could be misread as complete absence. | **Fixed in this work.** The gateway stamps receipt/freshness and idle transitions and enforces deployment identity; destructive boolean fields, inventory identity/resources and telemetry are validated strictly, and any malformed complete inventory fails closed as an invalid heartbeat. | `control_plane.py`, `registry.py`, `agent.py`, `models.py` |
| C12 | High | Placement could over-credit resources by ignoring node reservations/build reservations, double-charge routes already represented in inventory, trust invalid telemetry, or place disk requests without enforced quota. | **Fixed in this work.** Accounting unions live, create-reserved and build-reserved usage plus only unmatched route reservations; invalid/non-finite values fail safe; disk placement requires the quota capability; stale/incompatible nodes receive no capacity credit while still counting toward hard limits. | `control_plane.py`, `models.py`, `policy.py`, `agent.py` |
| C13 | Critical | Registry prune/GC could delete a manifest being pulled, built/pushed or used, delete another tag alias of the same digest, execute a stale plan, or overlap offline GC. | **Fixed in this work.** Durable route/build references, finite transient pull leases, digest-alias checks, execution-time revalidation and a cross-process maintenance fence order prune/use/build/GC. Protection is acquired before dispatch and matching confirmed completion/delete releases it. Ambiguous outcomes retain protection. | `managed_registry.py`, `control_plane.py`, `cli.py` |
| C14 | High | A long-lived sandbox registry lease can expire while the sandbox remains active if no renewal path runs. | **Fixed without renewal.** Route ownership is a non-expiring persistent reference, released only after a matching successful delete. A control-plane outage cannot expire a live route's protection; leaked references are conservative and require explicit reconciliation. | `control_plane.py`, `managed_registry.py` |
| C15 | High | Node JSON/image/build state could lose cross-process updates; builds could exceed the configured limit or remain “running” after process death. | **Fixed in this work.** Cross-process locking, coherent reservations, global build admission, drain-aware replay and startup failure marking are covered. | `images.py`, `sandbox.py`, `node_agent.py` |
| C16 | High | A container existing only in the runtime, but absent from the node state file, may not appear in proactive heartbeat inventory and therefore may consume resources invisibly until its ID is touched. | **Fixed for protocol-managed creates; mitigated for pre-existing/out-of-band containers.** The node persists a planned record before the runtime side effect, so heartbeat inventory and capacity include the operation through both crash windows; replay adopts a matching runtime object or resumes an absent one. There is still no general scan/adoption policy for containers created outside this protocol or before rollout. | `sandbox.py`, `node_agent.py` |
| C17 | High | A client bypassing the gateway could overcommit a node because aggregate sandbox admission is chiefly a gateway placement invariant. | **Fixed in this work for configured node capacity.** Under the cross-process create lock the node sums running and planned records and rejects CPU/memory/disk overflow before the runtime call; sync capacity includes configured overcommit and async parity uses its configured totals. Zero/unconfigured dimensions deliberately remain unbounded for compatibility. | `sandbox.py`, `node_agent.py`, `async_node_agent.py` |
| C18 | High | Malformed, negative, missing or chunked lengths and unbounded request bodies could desynchronize requests or exhaust memory; invalid policy/resource values could inflate capacity. | **Fixed in this work.** Node and control-plane request framing is strict and request sizes are capped; config, overcommit, accounting and non-finite metric inputs are validated/sanitized. Response buffering remains a performance issue below. | `control_plane.py`, `node_agent.py`, `config.py`, `models.py`, `agent.py` |
| C19 | Medium | Runtime delete failures could be reported as client validation errors or treated as successful absence. | **Fixed in this work.** Missing targets are idempotent success, validation conflicts remain 4xx, and transient runtime failures surface as retryable server errors without dropping state. | `sandbox.py`, `node_agent.py` |
| C20 | High | Model-relay leases could accept a response after expiry, duplicate an in-process claim, retain canceled work, or exceed global/per-rollout/byte admission. | **Fixed for the supported single relay process.** Expiry is rechecked at response commit, cancellation releases admission, rollout/request/lease identities fence stale work, and request/per-rollout/byte limits are enforced. Relay restart remains a visible request failure rather than persisting orphan work that can no longer reattach to the caller. | `model_relay.py`, `cli.py` |
| C21 | Medium | A full websocket/output queue could block command completion and leak sessions/history indefinitely. | **Fixed in this work.** Completion is non-blocking and event/session histories are bounded. | `async_exec.py`, `sandbox_exec.py` |
| C22 | Medium | Metrics/events could grow without bound or emit invalid JSON from oversized/non-finite payloads. | **Fixed in this work.** Metrics rotate, enforce event limits, replace oversized entries with markers and sanitize values. | `metrics.py`, `models.py` |
| C23 | Medium | Separate gateway and heartbeat trust channels could share a credential, allowing public-link/API credentials to impersonate a node heartbeat. | **Fixed in this work.** Generated deployments use an independent heartbeat credential and the gateway enforces channel-scoped tokens, with an explicit legacy fallback when omitted. | `control_plane.py`, `deploy.py`, `config.py` |
| C24 | High | A failed image pull/node create after a node was selected may clear the route without preserving a durable pending-demand signal, so capacity demand disappears unless the caller retries. | **Fixed locally; architectural queue remains.** Pre-dispatch registry failure and terminal image-pull failure persist an expiring demand record carrying the failed generation, operation ID, spec hash, resources and reason. Ambiguous node create keeps its durable route instead. Success/new allocation or explicit delete clears the signal. D9 remains because request acceptance is not yet an authoritative desired-state queue. | `control_plane.py`, `routing.py`, `cli.py` |
| C25 | High | The legacy one-shot reconcile command could mutate provider state through a second orchestration path. | **Removed.** `reconcile` is read-only; the sole mutating one-shot is `autoscaler-loop --once`, which uses the same local controller lock, journal, recovery and drain workflow as the recurring service. | `cli.py`, `reconcile.py`, `autoscaler_state.py` |
| C27 | High | Re-registering a reused relay rollout id allowed delayed unregister, poll, heartbeat, renew or completion traffic from the prior incarnation to affect the replacement rollout. | **Fixed in this work.** Each registration receives a random incarnation token. Replacement cancels prior work, and every rollout-scoped worker mutation is compare-and-set fenced in the single in-memory state machine. | `model_relay.py` |

## Performance and scalability

| ID | Severity | Failure mode | Current status | Relevant components |
|---|---|---|---|---|
| P1 | High | Gateway/node proxy and file responses are fully buffered; large uploads also require an in-memory request buffer. Concurrent large transfers can exhaust RAM and add latency. | **Mitigated** by admission-before-read, explicit input framing, 16 MiB JSON and 256 MiB file/proxy caps. Node Docker file reads now stop at limit+1 while bounding diagnostics, so a container file cannot first consume unbounded node RAM. **Remaining:** gateway streaming request/response forwarding and backpressure are not implemented. | `control_plane.py`, `node_agent.py`, `sandbox.py` |
| P2 | Medium | Every proxy/UCloud/registry operation opens a new HTTP connection; there is no client connection pool, multiplexing or shared retry budget. | **Remaining.** Timeouts bound stalls, but connection setup and socket churn scale with request and node count. | `control_plane.py`, `ucloud.py`, `managed_registry.py`, `agent.py` |
| P3 | High | Several local stores rewrite or scan O(N) JSON on each mutation (heartbeats, node sandbox/image state, registry usage, some metrics/session state). Lock hold time and write amplification grow with fleet/history size. | **Remaining**, with atomic files, interprocess locks, rotation and bounded histories reducing corruption/growth risk. | `registry.py`, `sandbox.py`, `images.py`, `managed_registry.py`, `metrics.py` |
| P4 | Medium | Route operations refreshed and copied the full SQLite routing state, making point reads and writes O(number of routes/signals) and creating a second source of truth. | **Fixed in this work for normal operations.** The in-memory mirror is removed and point operations use targeted SQL. Explicit full-state load and fleet reconciliation still materialize O(number of routes/signals) snapshots by design. | `routing.py`, `control_plane.py` |
| P5 | Medium | Full sandbox refresh/recovery can fan out to every ready node and scan every sandbox; create-time legacy discovery may also probe nodes. | **Mitigated.** Fresh complete heartbeat inventories skip create-time discovery entirely; legacy recovery has a two-second total fallback budget. Normal list is cache-first and refresh is explicit. Worst-case operator refresh remains O(nodes + sandboxes). | `control_plane.py`, `agent.py` |
| P6 | Medium | Exhaustive UCloud job pagination and label correlation are O(all relevant jobs) each recovery/reconcile cycle and can dominate large projects. | **Fixed for correctness, remaining as a cost.** Pagination is complete and cursor-safe; incremental provider inventory is not available in this client. | `ucloud.py`, `cli.py`, `autoscaler_state.py` |
| P7 | Medium | The stdlib thread-per-request servers can create substantial thread/socket overhead during bursts or retain threads indefinitely on slow clients. | **Mitigated** by a larger accept backlog, a 256-request thread bound, accepted-socket read timeouts, bounded concurrent sandbox creates and admission limits. An async/streaming gateway architecture remains future work. | `http_server.py`, `control_plane.py`, `node_agent.py` |
| P8 | High | SQLite/WAL serializes routing/provider writers and is suitable for local state, not arbitrary network filesystems or multi-region throughput. | **Remaining as an explicit topology boundary.** The generated deployment uses one control host and one relay process. Multi-host control or relay HA requires a server database/queue and an API that lets callers reattach or retry idempotently. | `autoscaler_state.py`, `routing.py`, `model_relay.py` |
| P9 | Medium | Registry catalog/tag/digest inspection is proportional to registry size and remote request count; prune execution can hold coordination while making bounded remote calls. | **Mitigated** with dashboard caching, usage-based retention, alias-aware planning and request timeouts. Large registries still need pagination/indexing/operational tuning. | `managed_registry.py`, `control_plane.py` |
| P10 | Medium | Repeated image pulls/builds and cold-cache placement can amplify network and registry load. | **Mitigated.** Heartbeats advertise cache summaries, concurrent cold pulls are de-duplicated, and preparation/warmup can spread images ahead of demand. | `control_plane.py`, `images.py`, `routing.py` |
| P11 | Medium | Observability and exec history could grow without bound and make every read slower. | **Fixed in this work.** Metrics rotate and bound event size/count; exec event/session retention is bounded. | `metrics.py`, `sandbox_exec.py` |

## Design and architecture

| ID | Severity | Failure mode | Current status | Relevant components |
|---|---|---|---|---|
| D1 | Critical | POSIX file locks and SQLite coordinate processes on one host/shared filesystem but do not provide a multi-host leader, linearizable placement or consensus. Split-brain gateways/autoscalers can violate single-writer assumptions if files/locks are not truly shared. | **Remaining architectural boundary.** Current locking is appropriate for the documented single-host topology. Multi-host HA needs a transactional consensus-capable store for routes, placement, leases and provider operations. | `routing.py`, `control_plane.py`, `autoscaler_state.py`, `managed_registry.py` |
| D2 | High | State is split among UCloud jobs, heartbeat JSON, routing SQLite, node JSON/runtime objects, autoscaler SQLite and registry usage JSON; there is no global transaction across those systems. | **Mitigated** by generations, tombstones, complete inventories, provider labels/journals and leases. Reconciliation remains a core architectural requirement rather than an exceptional repair path. | service-wide; `docs/distributed-state-protocol.md` |
| D3 | Critical | The node-agent HTTP API lacked an independent workload-facing authentication barrier. If bridged/host networking lets an untrusted sandbox reach the host API, it could invoke privileged create/delete/build/file operations. | **Fixed for generated deployments; rollout/trust-scope mitigation remains.** Every non-health sync/async node route can require a constant-time Bearer check; gateway, heartbeat self-fetch, autoscaler drain, warmup, VM init and direct clients use the distinct generated node-control credential while stripping public credentials. Legacy omission remains explicitly possible, and one deployment-wide token is not per-node identity. Firewalling and eventual per-node mTLS/authorization are still required before treating workloads as hostile. | `node_agent.py`, `async_node_agent.py`, `control_plane.py`, `deploy.py`, deployment/network configuration |
| D4 | Critical | Docker image builds execute untrusted Dockerfiles/build contexts with host Docker privileges, which is a host-compromise boundary rather than sandbox isolation. | **Remaining.** Builder-only VMs reduce blast radius but do not make an individual builder safe for mutually untrusted tenants. Use disposable hardened builders/rootless or remote build isolation plus provenance/policy controls. | `images.py`, `node_agent.py`, builder deployment |
| D5 | High | UCloud provider APIs do not share a transaction with local state; a controller crash cannot cancel a request already in flight. | **Mitigated.** Submission is journaled before the call and creates are correlated by immutable labels, so the next local controller does not blindly duplicate work. Permanently uncertain/conflicting operations still need an operator/runbook. | `autoscaler_state.py`, `ucloud.py`, `cli.py` |
| D6 | High | Registry protection can diverge from route/build truth after ambiguous outcomes or crashes. | **Mitigated fail-safe.** Routes and pushed builds use non-expiring references, so outages cannot expose live manifests to prune. Ambiguous/crashed asynchronous builds may leak protection until explicit terminal reconciliation; acquisition remains fail-closed. | `control_plane.py`, `managed_registry.py` |
| D7 | High | A relay process restart loses queued work and the caller connection; worker execution remains at-least-once after lease expiry. | **Explicit single-process boundary.** Persisting the queue locally was removed because no caller reattachment/idempotency API exists, so it preserved orphan work without restoring the request. Multi-host durability requires a server-backed broker plus caller retry/reattachment semantics. | `model_relay.py`, `cli.py`, relay deployment/configuration |
| D8 | High | The control plane combines routing, placement, reconciliation, image scheduling, registry coordination, metrics and dashboard serving in one large handler, increasing coupling and making independent scaling/failure isolation difficult. | **Remaining.** Durable store APIs and state protocols improve boundaries, but responsibilities are not yet separate services. | `control_plane.py`, `routing.py`, `metrics.py` |
| D9 | High | Demand is a mixture of expiring signals and request retries rather than a durable desired-state queue; expiration can trade accepted intent for cleanup. | **Remaining architectural boundary.** Failed pre-dispatch/pull demand now retains incarnation identity locally, and ambiguous create retains its route, but prepared/pending records remain autoscaling hints rather than an authoritative exactly-once work queue/outbox. | `routing.py`, `control_plane.py`, `cli.py` |
| D10 | High | Legacy generation-zero node operations coexist with the fenced protocol, so a partially upgraded deployment cannot claim every invariant. | **Mitigated.** New protocol operations fence legacy mutation of newer generations and all provider mutation entry points use the journal/drain workflow. Legacy node compatibility should be removed only after capability-gated rollout. | `sandbox.py`, `node_agent.py`, `cli.py`, `reconcile.py` |
| D11 | High | Drain safety spans node admission, heartbeat inventory, autoscaler planning and provider stop; partial deployment creates mixed semantics. | **Fixed in current recurring and one-shot controllers, with a rollout boundary.** The durable token/epoch handshake spans those components and recovery. Mixed-version capability gating remains necessary during deployment upgrades. | `sandbox.py`, `node_agent.py`, `agent.py`, `autoscaler_state.py`, `cli.py`, `reconcile.py` |
| D12 | Medium | Local state files contain sensitive tokens, sandbox specifications and operational metadata; filesystem permissions/backups become part of the security model. | **Mitigated.** Owner-only file modes and separate generated credentials exist in relevant paths. Relay prompt/response bodies are no longer persisted. Host access, backup encryption/retention and secret rotation remain operator responsibilities. | `deploy.py`, `config.py`, state directories |

## Practical readiness summary

The repository now has credible crash/retry protocols for sandbox generations,
node tombstones and runtime recovery, registry use/prune fencing, complete UCloud
pagination, gateway-stamped freshness/resource accounting, and journaled
autoscaler provider operations. Request/admission/history bounds also remove
several straightforward exhaustion and malformed-input failures.

It should still be treated as a **single-host control-plane architecture**. The
largest production gaps are multi-host consensus, mixed-version capability and
node-control/firewall rollout, per-node authorization, adoption/quarantine of
out-of-band runtime objects, authoritative desired-state semantics, isolated
untrusted builds, O(N) local JSON state costs, and streamed/server-backed relay
and proxy data paths. Persistent registry references, aggregate node admission,
provider journaling and bounded single-process relay behavior no longer depend
on those follow-up items.
